# UX Standards — sowhat 터미널 인터랙션

모든 워크플로우 파일이 준수해야 하는 UX 표준.

---

## 1. 선택지는 반드시 숫자 (CRITICAL)

모든 선택지는 `[1]`, `[2]`, `[3]`... 형식만 사용한다.
알파벳 선택지 `[A]`, `[B]`, `[C]`, `[D]`, `[M]`, `[S]` 등은 **절대 금지**한다.
테이블/표 안에 선택지를 나열하는 것도 금지한다.
워크플로우 템플릿에 정의된 포맷을 "더 나은" 형식으로 재구성하지 않는다.

```
# 올바른 예
[1] 승인 — settle 진행
[2] 수정 필요 — 어떤 부분?
[3] 건너뛰기 — 나중에 다시

# 금지 예
[A] 승인 — settle 진행
[M] 수정 필요 — 어떤 부분?
[S] 건너뛰기 — 나중에 다시
```

서브 질문에서도 동일하다. 숫자 연속성을 유지할 필요 없이 각 질문마다 `[1]`부터 시작한다.

### AskUserQuestion 사용 금지

**선택지를 제시할 때 AskUserQuestion 도구를 사용하지 않는다.**

텍스트에 `[1]` `[2]` `[3]`으로 선택지를 표시한 뒤, 유저의 자유 입력(번호 또는 텍스트)을 그대로 받는다.

AskUserQuestion은 자체 번호 체계를 만들어 텍스트에 표시한 번호와 불일치를 일으킨다:

```
# 문제 상황 — 절대 이렇게 하지 않는다

텍스트 표시:           AskUserQuestion 실제:
[1] 한국주택금융공사    1. PIR 15배          ← 불일치!
[2] PIR 15배           2. 생략
[3] 생략               3. Sub-Research
[4] 직접 작성           4. Type something
```

유저가 `1`을 입력하면 텍스트 기준 [1]을 선택한 것이다. AskUserQuestion의 재구성된 1번이 아니다.

**올바른 패턴:**
1. 텍스트로 선택지를 표시한다
2. 유저의 다음 메시지를 기다린다 (AskUserQuestion 호출 없이)
3. 유저가 번호를 입력하면 텍스트에 표시한 번호에 대응시킨다
4. 유저가 텍스트를 입력하면 자유 입력으로 처리한다

## 2. 터미널 안전 포맷팅

### 구분선

굵은 구분선 `━` 대신 일반 대시 `-`를 사용한다. 너비는 40자 고정.

```
# 올바른 예
----------------------------------------

# 금지 예
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 컨텍스트 배너

박스 문자(`┌─┐│└┘`) 대신 `>` 인용 블록을 사용한다.

```
# 올바른 예
> Thesis: "통합 비용 해소로 B2B SaaS 이탈을 막을 수 있다"
> 섹션: 01-market-size | 스텝 4/9 — Grounds
> Stasis: 사실 주장 | Scheme: statistics
> Claim: "국내 SaaS 시장은 연 28% 성장 중이다"

# 금지 예
┌─── 진행 컨텍스트 ──────────────────────────────────────┐
│ Thesis: "통합 비용 해소로 B2B SaaS 이탈을 막을 수 있다"  │
│ ...                                                     │
└────────────────────────────────────────────────────────┘
```

### 체크포인트 블록

박스 대신 헤더 + 인용 블록을 사용한다.

```
# 올바른 예
**[verify-argument]** 01-market-size

> 자동 검증 결과:
> ✅ Claim ↔ Thesis 정합성
> ✅ Grounds 존재
> ⚠️ Warrant: 암묵적

[1] 승인 — settle 진행
[2] 수정 필요 — 어떤 부분?
[3] 건너뛰기 — 나중에 다시

# 금지 예
┌─── verify-argument ───────────────────────────┐
│ ...                                            │
│ [A] 승인 — settle 진행                         │
│ [M] 수정 필요 — 어떤 부분?                     │
│ [S] 건너뛰기 — 나중에 다시                     │
└────────────────────────────────────────────────┘
```

### 섹션 구분 라벨

`──── 예시 ────` 대신 `**예시:**` 볼드 텍스트를 사용한다.

```
# 올바른 예
**예시:**
  "국내 SaaS 시장은 연 28% 성장 중이다"

**제안:**
  [1] {제안 1}
  [2] {제안 2}

# 금지 예
──── 예시 ────
  "국내 SaaS 시장은 연 28% 성장 중이다"

──── 제안 ────
  [1] {제안 1}
```

## 3. 다음 액션 안내 (Flow Navigation)

### 커맨드 완료 후

`▶ 다음` 섹션 대신, 번호 선택지로 다음 액션을 제시한다.
유저가 커맨드를 직접 타이핑할 필요 없이 번호만 입력하면 된다.

```
# 올바른 예
----------------------------------------
다음 액션:

[1] Thesis 확정 (settle)
[2] 섹션 전개 시작 (expand 01)
[3] 전체 상태 확인 (progress)
----------------------------------------

# 금지 예
## ▶ 다음

**00-thesis: Thesis 확정** — Thesis를 settled로 전환

`/sowhat:settle thesis`

<sub>`/clear` 후 실행 → 컨텍스트 초기화</sub>
```

유저가 번호를 입력하면 Claude가 해당 `/sowhat:` 커맨드를 직접 실행한다.
커맨드 이름은 괄호 안에 간략히 표기하되, 유저가 타이핑할 필요는 없다.

### progress/resume 출력 후

동일하게 번호 선택지로 제시한다.

## 4. 컨텍스트 브레드크럼

깊은 워크플로우에서 현재 위치를 한 줄로 표시한다.
모든 핑퐁 질문 앞에 컨텍스트 배너와 함께 표시한다.

```
init > SCQ 핑퐁 > Question
expand 01-market > 스텝 4/9 Grounds > 근거 유형 선택
debate 02-tech > Round 2/3 > Pro 방어
spec 05-func-req > 기능 3/8
```

배너의 첫 줄에 브레드크럼을 배치한다:

```
> [expand 01-market > 스텝 4/9 Grounds > 근거 유형]
> Thesis: "통합 비용 해소로 B2B SaaS 이탈을 막을 수 있다"
> Claim: "국내 SaaS 시장은 연 28% 성장 중이다"
```

## 5. 질문 포맷 — 호흡과 집중

### 원칙: 3-블록 분리

모든 핑퐁 질문은 **배너 → 질문 → 선택** 3블록으로 나누고, 블록 사이에 빈 줄을 넣어 호흡을 준다.

### 블록 1: 컨텍스트 배너 (현재 위치 + 집중 대상)

```
> [expand 01-market > 스텝 4/9 Grounds]
> Claim: "국내 SaaS 시장은 연 28% 성장 중이다"
```

**규칙:**
- 브레드크럼 + **현재 집중해야 할 필드값**만 표시
- Thesis는 생략 (이미 알고 있음 — 첫 질문에서만 1회 표시)
- Claim이 가장 중요 — 매 질문마다 반복하여 맥락 유지
- 2줄 이하로 유지

### 블록 2: 질문 (빈 줄로 분리)

```

❓ 이 주장을 지지하는 근거의 유형은?

  예) 수치/데이터 → 출처, 규모, 시기를 묻겠습니다

```

**규칙:**
- 질문 위아래에 빈 줄 1개씩 — 시각적 호흡
- 예시는 질문 바로 아래에 `예)` 접두어로 간결하게
- `**예시:**` 볼드 라벨 불필요 — `예)` 한 줄이면 충분
- 예시가 여러 개면 각각 `예)` 줄로 나열

### 블록 3: 선택지 (빈 줄로 분리)

```

[1] 수치/데이터
[2] 인터뷰/설문
[3] 사례/비교
[4] 전문가 의견
[5] 직접 서술
[6] Sub-Research (인라인 검색)
```

**규칙:**
- 선택지 위에 빈 줄 1개
- `**선택:**` `**제안:**` 라벨 불필요 — 번호가 곧 선택지임을 알려줌
- 선택지 간 빈 줄 없이 밀집 (하나의 덩어리로 스캔)

### 전체 조립 예시

```
> [expand 01-market > 스텝 4/9 Grounds]
> Claim: "국내 SaaS 시장은 연 28% 성장 중이다"

❓ 이 주장을 지지하는 근거의 유형은?

  예) 수치/데이터 → 출처, 규모, 시기를 묻겠습니다

[1] 수치/데이터
[2] 인터뷰/설문
[3] 사례/비교
[4] 전문가 의견
[5] 직접 서술
[6] Sub-Research (인라인 검색)
```

### 답변 확인 후 다음 질문 전

유저가 답변하면 **확인 1줄 + 빈 줄 + 다음 질문** 패턴:

```
✓ Grounds 기록됨: IDC 2024 리포트, CAGR 27.8%

> [expand 01-market > 스텝 5/9 Warrant]
> Claim: "국내 SaaS 시장은 연 28% 성장 중이다"
> Grounds: IDC 2024, CAGR 27.8%

❓ Grounds가 Claim을 왜 지지하는지, 논리적 연결은?

  예) "CAGR 27%+ 시장은 신규 진입자에게 충분한 성장 여력이 있다"

[1] 위 예시 채택
[2] 직접 작성
[3] 잘 모르겠다 → Open Question 등록
```

### 금지 패턴

```
# 나쁜 예 — 빽빽하고 집중 대상이 불명확

> [expand 01-market > 스텝 4/9 Grounds]
> Thesis: "통합 비용 해소로 B2B SaaS 이탈을 막을 수 있다"
> 이 섹션 논거: "시장 규모가 충분히 크다"
> Stasis: 사실 주장 | Scheme: statistics
> Claim: "국내 SaaS 시장은 연 28% 성장 중이다"
❓ 이 주장을 지지하는 근거의 유형은?
**예시:**
  수치/데이터 → 출처, 규모, 시기를 묻겠습니다
**선택:**
  [1] 수치/데이터
  [2] 인터뷰/설문
```

위 나쁜 예의 문제:
- 배너 5줄 — 읽기 전에 지침
- Thesis/논거/Stasis/Scheme — 이미 아는 정보가 매번 반복
- 질문과 선택지 사이 빈 줄 없음 — 한 덩어리로 보임
- `**예시:**` `**선택:**` 라벨이 공간 차지

## 6. 요약 규칙

| 항목 | 규칙 |
|------|------|
| 선택지 | 숫자만 `[1]` `[2]` `[3]`... |
| 입력 수집 | AskUserQuestion 금지. 텍스트 표시 후 자유 입력 대기 |
| 질문 구조 | 3-블록 분리: 배너 → (빈줄) → 질문+예시 → (빈줄) → 선택지 |
| 배너 | 브레드크럼 + Claim만. 2줄 이하. Thesis는 첫 질문에서만 |
| 예시 | `예)` 접두어 1줄. `**예시:**` 라벨 불필요 |
| 선택 라벨 | `**선택:**` `**제안:**` 불필요. 번호가 곧 선택지 |
| 답변 확인 | `✓` 1줄 확인 → 빈 줄 → 다음 질문 |
| 구분선 | `-` 40자: `----------------------------------------` |
| 컨텍스트 | `>` 인용 블록 |
| 체크포인트 | `**[type]** name` + 인용 블록 |
| 다음 액션 | 번호 선택지 (커맨드 타이핑 불필요) |
| 브레드크럼 | `> [command > step > substep]` |
| 박스 문자 | 전면 금지 (`┌─┐│└┘━`) |
| 이모지 | 상태 이모지만 허용 (✅❌⚠️❓✓🔍) |
