# UX Standards — sowhat 터미널 인터랙션

모든 워크플로우 파일이 준수해야 하는 UX 표준.

---

## 1. 선택지는 반드시 숫자

모든 선택지는 `[1]`, `[2]`, `[3]`... 형식만 사용한다.
알파벳 선택지 `[A]`, `[M]`, `[S]` 등은 **금지**한다.

```
# 올바른 예
[1] 승인 — settle 진행
[2] 수정 필요 — 어떤 부분?
[3] 건너뛰기 — 나중에 다시

# 금지 예
[A] 승인 — settle 진행
[M] 수정 필요 — 어떤 부분?
[S] 건너뛰기 — 나중에 다시
```

서브 질문에서도 동일하다. 숫자 연속성을 유지할 필요 없이 각 질문마다 `[1]`부터 시작한다.

---

## 2. 터미널 안전 포맷팅

### 구분선

굵은 구분선 `━` 대신 일반 대시 `-`를 사용한다. 너비는 40자 고정.

```
# 올바른 예
----------------------------------------

# 금지 예
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 컨텍스트 배너

박스 문자(`┌─┐│└┘`) 대신 `>` 인용 블록을 사용한다.

```
# 올바른 예
> Thesis: "통합 비용 해소로 B2B SaaS 이탈을 막을 수 있다"
> 섹션: 01-market-size | 스텝 4/9 — Grounds
> Stasis: 사실 주장 | Scheme: statistics
> Claim: "국내 SaaS 시장은 연 28% 성장 중이다"

# 금지 예
┌─── 진행 컨텍스트 ──────────────────────────────────────┐
│ Thesis: "통합 비용 해소로 B2B SaaS 이탈을 막을 수 있다"  │
│ ...                                                     │
└────────────────────────────────────────────────────────┘
```

### 체크포인트 블록

박스 대신 헤더 + 인용 블록을 사용한다.

```
# 올바른 예
**[verify-argument]** 01-market-size

> 자동 검증 결과:
> ✅ Claim ↔ Thesis 정합성
> ✅ Grounds 존재
> ⚠️ Warrant: 암묵적

[1] 승인 — settle 진행
[2] 수정 필요 — 어떤 부분?
[3] 건너뛰기 — 나중에 다시

# 금지 예
┌─── verify-argument ───────────────────────────┐
│ ...                                            │
│ [A] 승인 — settle 진행                         │
│ [M] 수정 필요 — 어떤 부분?                     │
│ [S] 건너뛰기 — 나중에 다시                     │
└────────────────────────────────────────────────┘
```

### 섹션 구분 라벨

`──── 예시 ────` 대신 `**예시:**` 볼드 텍스트를 사용한다.

```
# 올바른 예
**예시:**
  "국내 SaaS 시장은 연 28% 성장 중이다"

**제안:**
  [1] {제안 1}
  [2] {제안 2}

# 금지 예
──── 예시 ────
  "국내 SaaS 시장은 연 28% 성장 중이다"

──── 제안 ────
  [1] {제안 1}
```

## 3. 다음 액션 안내 (Flow Navigation)

### 커맨드 완료 후

`▶ 다음` 섹션 대신, 번호 선택지로 다음 액션을 제시한다.
유저가 커맨드를 직접 타이핑할 필요 없이 번호만 입력하면 된다.

```
# 올바른 예
----------------------------------------
다음 액션:

[1] Thesis 확정 (settle)
[2] 섹션 전개 시작 (expand 01)
[3] 전체 상태 확인 (progress)
----------------------------------------

# 금지 예
## ▶ 다음

**00-thesis: Thesis 확정** — Thesis를 settled로 전환

`/sowhat:settle thesis`

<sub>`/clear` 후 실행 → 컨텍스트 초기화</sub>
```

유저가 번호를 입력하면 Claude가 해당 `/sowhat:` 커맨드를 직접 실행한다.
커맨드 이름은 괄호 안에 간략히 표기하되, 유저가 타이핑할 필요는 없다.

### progress/resume 출력 후

동일하게 번호 선택지로 제시한다.

## 4. 컨텍스트 브레드크럼

깊은 워크플로우에서 현재 위치를 한 줄로 표시한다.
모든 핑퐁 질문 앞에 컨텍스트 배너와 함께 표시한다.

```
init > SCQ 핑퐁 > Question
expand 01-market > 스텝 4/9 Grounds > 근거 유형 선택
debate 02-tech > Round 2/3 > Pro 방어
spec 05-func-req > 기능 3/8
```

배너의 첫 줄에 브레드크럼을 배치한다:

```
> [expand 01-market > 스텝 4/9 Grounds > 근거 유형]
> Thesis: "통합 비용 해소로 B2B SaaS 이탈을 막을 수 있다"
> Claim: "국내 SaaS 시장은 연 28% 성장 중이다"
```

---

## 5. 질문 포맷

```
> [expand 01-market > 스텝 4/9 Grounds]
> Thesis: "통합 비용 해소로..."
> Claim: "국내 SaaS 시장은..."

❓ 이 주장을 지지하는 근거의 유형은?

**예시:**
  수치/데이터 → 출처, 규모, 시기를 묻겠습니다

**선택:**
  [1] 수치/데이터
  [2] 인터뷰/설문
  [3] 사례/비교
  [4] 전문가 의견
  [5] 직접 서술
  [6] Sub-Research (인라인 검색)
```

## 6. 요약 규칙

| 항목 | 규칙 |
|------|------|
| 선택지 | 숫자만 `[1]` `[2]` `[3]`... |
| 구분선 | `-` 40자: `----------------------------------------` |
| 컨텍스트 | `>` 인용 블록 |
| 체크포인트 | `**[type]** name` + 인용 블록 |
| 섹션 라벨 | `**라벨:**` 볼드 |
| 다음 액션 | 번호 선택지 (커맨드 타이핑 불필요) |
| 브레드크럼 | `> [command > step > substep]` |
| 박스 문자 | 전면 금지 (`┌─┐│└┘━`) |
| 이모지 | 상태 이모지만 허용 (✅❌⚠️❓🔍) |
